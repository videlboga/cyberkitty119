# План: параллельная обработка задач в Transkribator

## 1. Цели и ограничения
- Разделить обработку тяжёлых медиафайлов и диалоговую работу бота, чтобы несколько пользователей могли одновременно писать, загружать и получать ответы.
- Исключить блокировки основного event-loop Telegram-бота.
- Сохранить существующую бизнес-логику (транскрибация, форматирование, Drive/Sheets), минимально меняя интерфейс для пользователя.

## 2. Архитектура решения
- **Очередь заданий**
  - Новый модуль (например, `transkribator_modules/jobs/queue.py`) с интерфейсом `enqueue_job(payload)` / `fetch_next_job()`.
  - В первой итерации можно использовать Postgres (таблица `processing_jobs`) с полями: `id`, `user_id`, `type`, `payload_json`, `status`, `progress`, `created_at`, `started_at`, `finished_at`, `error`.
  - Дальше легко заменить на Redis/RabbitMQ, если понадобится.
- **Воркеры**
  - Отдельный процесс/скрипт `job_worker.py`, который в цикле забирает задания и выполняет шаги: скачивание → подготовка аудио → транскрибация → форматирование → автосаммори/индекс → отправка результатов по Telegram.
  - Количество воркеров настраивается переменной окружения; каждый воркер работает с одним заданием, что даёт параллелизм.
- **Основной бот**
  - В ветке получения медиафайла создаёт запись в очереди, отвечает пользователю «Запустил обработку» и выходи из handler’а без долгих вычислений.
  - Поддерживает короткие ответы (вопросы, поиск, правки текста) как сейчас, пользуясь автосаммори и обновлёнными правилами.
- **Обновление статуса**
  - Воркеры периодически обновляют поле `progress` и отправляют промежуточные сообщения через Telegram API (например, после скачивания и после транскрибации).
  - При ошибках воркер отмечает задание как `failed`, логирует стек и уведомляет пользователя.

## 3. Изменения по модулям
1. **Новая таблица/миграция**
   - Файл Alembic с таблицей `processing_jobs`.
   - DAO/сервис в `transkribator_modules/db` для работы с заданиями (создание, выборка на обработку с блокировкой, обновление статуса).
2. **Модуль очереди**
   - `jobs/queue.py` — функции `enqueue_media_job(user, note, file_info)` и `dequeue_job(worker_id)`.
   - Конфигурируемый таймаут повторного захвата (чтобы задание не потерялось, если воркер упал).
3. **Воркеры**
   - Скрипт `job_worker.py`: инициализация приложения, вечный цикл `dequeue -> process -> ack`.
   - Использует существующие утилиты (скачивание из Telegram, `_auto_finalize_note`, интеграцию с Drive и т. д.).
   - Пайплайн задач разбит на стадии (подготовка/загрузка/транскрибация/финал/доставка/cleanup) с возможностью переопределения сервисов.
   - Поддерживается режим dry-run (`--dry-run`), который позволяет тестировать пайплайн без изменения статуса задач.
   - Для отладки можно ограничить количество задач (`--max-jobs` / `JOB_MAX_JOBS`), после чего воркер завершится автоматически.
   - Пустая очередь использует экспоненциальный backoff (`JOB_BACKOFF_MIN`/`JOB_BACKOFF_MAX` или CLI флаги), снижая нагрузку на БД.
4. **Изменения в бот-хендлерах**
   - В `handlers/entrypoint.py` при получении аудио/видео вместо синхронного процесса вызываем `enqueue_media_job`.
   - Отправляем пользователю мгновенный ответ «Принял файл, обработаю в фоне» + подвинутые обновления статуса (через воркера).
5. **Нотификации прогресса**
   - Общая функция `notify_user(job, message, *, edit=False)` чтобы и бот, и воркеры могли безопасно редактировать сообщения.
6. **Конфигурация**
   - Добавить переменные окружения: `JOB_QUEUE_BACKEND`, `JOB_WORKER_CONCURRENCY`, `JOB_POLL_INTERVAL`, таймауты.
   - В docker-compose — сервис для воркеров (например, `transkribator-job-worker` с указанием количества процессов).
   - Сервисы пайплайна (`transkribator_modules/jobs/services.py`) можно переопределить через сборку контекста — добавить фабрику, читающую типы из конфигурации (в планах).
   - `job_worker.py` поддерживает `--service-overrides` (или `MEDIA_SERVICE_OVERRIDES`), чтобы подменять реализацию стадий без пересборки; пример: `transkribator_modules.jobs.examples.simple_overrides:build` (см. `docs/worker_configuration.md`).

## 4. План внедрения по шагам
1. **Инфраструктура данных**
   - Создать таблицу `processing_jobs` + ORM сервис.
   - Подготовить тесты для enqueue/dequeue (`tests/test_job_queue.py`).
2. **Инкапсуляция текущей логики медиапайплайна**
   - Вынести текущий код обработки медиа в функцию `process_media_job(job)` (реиспользует существующие функции).
   - Адаптировать под асинхронную/синхронную модель воркера.
3. **Реализация воркера**
   - `python job_worker.py` — обрабатывает задания, обновляет прогресс, пушит сообщения.
   - Добавить graceful shutdown, логирование, метрики (при возможности).
4. **Изменение Telegram-хендлеров**
   - Обновить `process_text` и связанные хендлеры: при медиафайлах создаётся задание и ответ «обрабатываю».
   - Обработка текстовых запросов остаётся без очереди (они быстрые).
5. **Статусы и уведомления**
   - Публичный API воркера (например, через `Bot API sendMessage/editMessageText`) для прогресса: «скачал», «транскрибирую», «форматирую», «готово».
   - Учесть, что Telegram сообщение должно обновляться из воркера (нужен токен бота).
6. **Развёртывание**
   - Добавить новый сервис в docker-compose (в проде и дев-сборке).
   - Обновить `deployment`/`README` с инструкциями запуска воркеров.
7. **Тестирование**
   - Модульные тесты на enqueue/dequeue + интеграционный сценарий (используя мок Telegram API).
   - Локальный прогон: параллельно отправить несколько видео и убедиться в отсутствия блокировок.
   - Для новых пайплайновых стадий добавлен каркас тестов (`tests/test_media_pipeline.py`), запускать через `pytest` после установки зависимостей.

## 5. Риски и TODO
- Убедиться, что воркер перехватывает все исключения и не оставляет задания в `in_progress` навсегда.
- Следить за лимитами Telegram API (много сообщений о прогрессе — сделать `throttle`).
- В будущем можно перейти на Redis/RabbitMQ для очереди, если Postgres окажется узким местом.
