# Архитектура V2: Персональная память (Second Brain) & Headless Core

Данный документ описывает целевую архитектуру проекта, к которой мы переходим в рамках глобального рефакторинга. Система эволюционировала из простого Telegram-бота для транскрибации аудио в полноценный SaaS-продукт класса **«Персональная память» (Second Brain) с AI-агентом**.

## 1. Концепция: Domain-Driven Design (DDD)

Мы отходим от технического (слоистого) разделения кода в пользу предметно-ориентированного. Архитектура отражает суть продукта как когнитивной системы:

### 🌟 Домен 1: Ingestion (Органы чувств / Захват данных)
Отвечает за превращение «шума» из внешнего мира (аудио, видео, в перспективе — текст, форварды писем) в структурированные воспоминания.
*   **Transcriber:** Захват медиа, выделение голоса (Whisper, DeepInfra, локальный GPU-пайплайн).
*   **Structurizer (Осмысление):** Прогон сырого текста через LLM для создания читаемой выжимки, выделения сущностей и автоматического тегирования.

### 🧠 Домен 2: Memory (Память)
Надежное хранилище распарсенных данных и инструмент для поиска смыслов.
*   **Storage:** Реляционная БД (PostgreSQL) для метаданных, текстов, связей и папок.
*   **Semantic Search:** Хранилище эмбеддингов для умного гибридного/векторного поиска.
*   **Analytics:** Сбор метрик по обращениям к базе памяти.

### 🗣 Домен 3: Agent (Интерфейс мышления / Когнитивный движок)
Главный инструмент взаимодействия пользователя с его базой знаний. Общение естественным языком.
*   **Agent Runtime:** Цикл рассуждения (ReAct), выбор инструментов, обращение к LLM (OpenRouter).
*   **Session Context:** Управление состоянием текущего диалога (Redis).
*   **Agent Tools (Навыки):** Изолированные утилиты (например, `search_memory`, `focus_on_memory`), которые дергают внутренние сервисы Ядра для извлечения ответов из Памяти.

### ⚙️ Домен 4: System (Инфраструктура)
Базовый функционал SaaS.
*   **Auth & Users:** Единая аутентификация (Service Tokens для ботов, JWT/подобное для Web).
*   **Billing & Limits:** Тарифы, квоты минут, интеграция с ЮKassa.

## 2. API-First (Headless) подход

Бэкенд ничего не знает о том, откуда к нему пришел пользователь. Вся бизнес-логика существует **исключительно** внутри Core API (Ядра).

*   **Core API (Ядро на FastAPI):** Предоставляет унифицированные REST-эндпоинты (`/api/v1/ingest`, `/api/v1/memory`, `/api/v1/agent`). Не хранит стейт в оперативной памяти (Stateless) — всё состояние сессий пишется в Redis/БД, что позволяет легко масштабировать API.
*   **Workers (Асинхронные воркеры):** Отдельные процессы (например, транскрибация на GPU), которые берут тяжелые задачи из очереди и возвращают результат в Ядро.
*   **Тонкие клиенты (Thin Clients):** 
    *   *Telegram Bot / MAX Bot:* Слушают вебхуки платформ, парсят текст/файлы и отправляют простые HTTP REST-запросы к Core API.
    *   *Mini Apps / Web UI:* Обращаются к тем же эндпоинтам Core API.

## 3. Целевая структура репозитория (монорепа)

```text
cyberkitty119/
├── core_api/                     # 🧠 ГЛАВНЫЙ БЕКЕНД
│   ├── api/v1/                   # Внешние интерфейсы 
│   │   ├── ingest.py             # POST /ingest/audio (загрузить воспоминание)
│   │   ├── memory.py             # GET /memory (список), POST /memory/search
│   │   ├── agent.py              # POST /agent/chat (Главный эндпоинт общения!)
│   │   └── system.py             # Auth, Billing, Limits
│   │
│   ├── domains/                  # --- БИЗНЕС-ЛОГИКА ПРОДУКТА ---
│   │   ├── ingestion/            # 👂 Превращение шума в знания (Whisper, LLM-структуризация)
│   │   ├── memory/               # 🧠 Хранение и семантический поиск (PG, Qdrant/PGVector)
│   │   └── agent/                # 🗣 Умный помощник (ReAct движок, Промпты)
│   │       └── tools/            # Навыки агента (вызывают слои memory и system)
│   │
│   ├── system/                   # Авторизация, биллинг, интеграции
│   └── main.py                   # Запуск FastAPI
│
├── workers/                      # ⚙️ ТЯЖЕЛЫЕ ЗАДАЧИ
│   └── transcribe_worker.py      # Обработчик очереди локального Whisper/GPU
│
├── clients/                      # 🤖 ЛИЦО ПРОЕКТА (Тонкие клиенты)
│   ├── telegram_bot/             # Изолированный aiogram бот (без БД!)
│   ├── max_bot/                  # Изолированный MAX бот (без БД!)
│   └── web_ui/                   # Web-клиент (MiniApp)
│
└── docker/                       # 🐳 Контейнеризация по ролям
```

## 4. План плавной миграции (Roadmap)

Рефакторинг не должен останавливать развитие продукта или ломать текущий функционал в проде. Мы двигаемся итеративно:

1.  **Создание скелета Ядра:** Создать папку `core_api/api/v1/` и перенести туда текущие роуты из `api_server.py` и `miniapp.py` (Users, Billing, текущий API транскрибации). Свести всё в единый FastAPI инстанс.
2.  **Выделение бизнес-логики (Domains):** Избавление роутеров от "толстого" кода (SQLAlchemy, парсинг логики). Создание чистых сервисов в `domains/memory/` и `domains/ingestion/`.
3.  **Перенос Агента:** Перевод кода из `transkribator_modules/beta` в структуру `core_api/domains/agent/`. Подключение инструментов агента к новым чистым сервисам Ядра.
4.  **Унификация Авторизации и Стейта:** Настройка проверки токенов (боты / UI). Вынос `_ACTIVE_SESSIONS` из памяти ботов в Redis.
5.  **"Обезжиривание" Ботов:** Удаление прямых импортов `db` и бизнес-логики из `cyberkitty_modular.py` и `max_bot/native_handlers.py`. Переписывание их обработчиков на банальные `httpx.post()` к новому Ядру.
6.  **Изоляция микросервисов:** Разделение Docker-образов (Легкий образ для ботов, отдельный для Ядра, тяжелый для GPU-воркера).

## 5. Статус v2 плана (12 апреля 2026)

### Скелет Core API есть:
* Единый FastAPI уже монтирует `system/agent/memory/ingest/internal_bot` роутеры и может жить отдельно от клиентов (`core_api/main.py`).
* Доменные слои заведены, но роутеры всё ещё напрямую импортируют legacy ORM/репозитории вместо чистых сервисов, поэтому Step 2 закрыт лишь частично (`core_api/api/v1/system.py`).
* Код агента перенесён в `core_api/domains/agent`, однако бот вызывает LLM напрямую через этот пакет, минуя `/api/v1/agent`, так что перенос логики общения не завершён (`bot/handlers.py`).
* Проверка сервисных токенов и DI-хелперы для Redis/PG уже есть (`core_api/api/v1/dependencies.py`), но Telegram-бот по-прежнему держит активные QA/поисковые сессии в локальных `_ACTIVE_*` структурах вместо Redis, обозначая незавершённый Step 4.
* «Обезжиривание» бота телеграм частично готово: инъекция всех REST-запросов в Core API реализована (`bot/core_api_client.py`), но в коде остались прямые импорты серверных модулей, а Max-бот всё ещё говорит с БД напрямую (`max_bot/db.py`).
* Compose-стек разделяет роли (Bot API, Core API, воркер, Whisper) и у бота есть облегчённый Dockerfile (`Dockerfile.bot-v2`), но воркер продолжают собирать из старого монолита, а каталога `workers/` из плана пока нет.

### Готовность ботов
* **Telegram v2** уже использует чистый httpx-клиент: загрузка медиа, профиль, промокоды и поиск дергают `/api/v1/ingest|system|memory`, а служебные операции идут только через `/api/v1/internal_bot`.
* Само приложение держит только оркестрацию Telegram и может запускаться отдельно, что соответствует «тонкому клиенту».
* Для обратной совместимости написан отдельный `internal_bot`-роутер, который ходит в старые SQLAlchemy модели.
* **Gap:** агентский Q&A и управление состоянием заметок всё ещё исполняются внутри бота (свои `_ACTIVE_QA_SESSIONS`), потому Core API пока не знает о диалогах.
* **Max-бот** мигрирован лишь наполовину: он по-прежнему создаёт пользователей и задачи через внутренние модули, хотя прогресс опрашивает через Core API.

### End-to-end тест
1. Подготовьте `.env` с ключами `BOT_TOKEN`, `TELEGRAM_API_ID/HASH`, переменными БД и одинаковым `CORE_API_SERVICE_TOKEN`, затем выполните миграции `alembic upgrade head`.
2. Поднимите инфраструктуру: `sudo docker compose -f docker-compose.bot-v2.yml up --build postgres telegram-bot-api core-api worker real-whisper` (описаны сервисы и сети).
3. Проверите Core API через `curl -H 'X-Service-Token: <token>' http://localhost:8000/api/v1/system/health` и вызов `/api/v1/internal_bot/ensure_user`.
4. Убедитесь, что Telegram Bot API жив: запустите `python -m bot.test_local_api` и выполните `/start`, затем остановите и запустите основной бот `python -m bot.main`.
5. Отправьте аудио/voice сообщение: бот скачает файл, вызовет `/ingest/media`, а прогресс можно отслеживать через `GET /api/v1/internal_bot/jobs/{job_id}`.
6. Для Max-пути поднимите `uvicorn max_bot.server:app --reload`, шлите синтетические JSON ивенты `POST /webhook/max`, проверяя, что задачи создаются, прогресс ходит в Core, и ответы возвращаются.
