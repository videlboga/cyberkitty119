## Обзор связи планов

Календарный план выполнения НИОКР (грант Старт, 5 млн. руб.) охватывает 12 месяцев разработки и определяет 6 макроэтапов. План миграции на Queue + Worker архитектуру (600 часов) является **техническим детализированием фаз 1–6** этого календарного плана и фокусируется на инфраструктуре, интеграции компонентов и финализации системы.

## Детальное соответствие по месячным интервалам

### Месячный интервал 1–2: НИОКР Этап 1 (Аналитика & проектирование)
- **НИОКР фокус**: Анализ пользовательских сценариев, формализация проблемы, проектирование agent-first архитектуры, требования к памяти и действиям, протоколы испытаний.
- **Миграция**: Подготовительная фаза (не входит в текущие TASK-12..22)
  - ✅ ADR написан (коммит e2c9d62)
  - ✅ Контракт очереди зафиксирован
  - ✅ Требования к retries, backoff, dedupe определены
- **Результат**: Документированная архитектура, готовность к реализации

---

### Месячный интервал 3–4: НИОКР Этап 2 (Базовый агентный прототип)
- **НИОКР фокус**: Базовый agent flow, интеграция автоматической транскрипции, первичное интеллектуальное структурирование, базовая долговременная память, семантический поиск.
- **Миграция**: **Фаза 1 — Infrastructure & Queue Foundation**
  - **TASK-12** (16h): Database Architecture & ADR — зафиксировать схему jobs table, миграция Alembic
  - **TASK-13** (80h): Job Queue Service Layer — DAO, services, retry logic, unit + integration тесты
  - **TASK-14** (110h): Worker Service & Whisper — FastAPI service, Dockerfile, entrypoint, benchmarking
  - **Subtotal**: 206h
- **Результат**: Функциональный queue + worker; основной E2E smoke (audio → transcript → DB update)
- **Acceptance**: Docker-compose поднимает стек; worker обрабатывает задачи асинхронно

---

### Месячный интервал 5–6: НИОКР Этап 3 (Развитие агентной логики)
- **НИОКР фокус**: Управляющий контур качества, механизмы выбора стратегии, оптимизация устойчивости, экспериментальная оценка характеристик.
- **Миграция**: **Фаза 2 — Core Features Integration**
  - **TASK-15** (60h): Transcriber Pipeline v4 & LLM Formatting — chunking, OpenRouter integration, fallback
  - **TASK-16** (140h): Bot Handlers Refactor & Deduplication — processing_guard, update_dedupe, event logging, large file handling
  - **TASK-17** (40h): Payment Flow & Monitoring — Yukassa webhook, sandbox, monitoring
  - **Subtotal**: 240h
- **Результат**: Полнофункциональный bot + payment flow; улучшенная обработка ошибок и dedupe логика; надёжность пайплайна
- **Acceptance**: Основные user flows работают (start → upload → transcribe → format → pay); dedupe предотвращает дубли

---

### Месячный интервал 7–8: НИОКР Этап 4 (Агентные действия & интеграции)
- **НИОКР фокус**: Автоматическое извлечение задач, интеграция с календарями и системой задач, расширение памяти, доработка интерфейса.
- **Миграция**: **Фаза 3 — Deployment, Testing & CI**
  - **TASK-18** (40h): Docker & Deployment Configuration — оптимизация образов, docker-compose, build strategy, документация deploy
  - **TASK-19** (32h): Tools, Scripts & Documentation — minimal_app, dev utilities, примеры использования
  - **TASK-20** (40h): Testing & CI Pipeline — unit + integration тесты, CI в GitHub Actions, фиксирование flaky тестов
  - **Subtotal**: 112h
- **Результат**: Reproducible containerized deployment; высокое качество кода (тесты + lint); готовость к staging/production
- **Acceptance**: CI pipeline проходит; все базовые тесты зелёные; docker compose поднимается одной командой

---

### Месячный интервал 9–10: НИОКР Этап 5 (Экспериментальная валидация)
- **НИОКР фокус**: Комплексная проверка количественных характеристик, оптимизация качества и производительности, анализ устойчивости.

- **Результат**: Production-ready code; очищенный репозиторий; готовность к финальному релизу и мониторингу

## Риски и точки синхронизации

### Критические зависимости

1. **Database Migration (TASK-12)**
   - **Риск**: Изменение схемы на production может потребовать downtime
   - **Синхронизация**: Координация с ops; план миграции должен быть согласован перед production deployment

2. **Bot Refactor (TASK-16, ~140h)**
   - **Риск**: Самая объёмная задача; высокая вероятность регрессий
   - **Синхронизация**: Требует comprehensive smoke тестирования; отложить финализацию до TASK-20 (CI pipeline)

3. **Payment Integration (TASK-17)**
   - **Риск**: Критичные для монетизации; ошибки могут привести к потере доходов
   - **Синхронизация**: Использовать sandbox; проверить все сценарии перед production

4. **CI Pipeline Setup (TASK-20)**
   - **Риск**: Если не настроить своевременно, баги будут найдены поздно
   - **Синхронизация**: Начать рано (TASK-20 может быть параллельна TASK-19)

## Выводы и рекомендации

### ✅ Что совпадает

- **Месячные интервалы**: План миграции (февраль 2026 = месяц 12) синхронизирован с календарным планом НИОКР (начало ноябрь 2025 = месяц 1)
- **Критические компоненты**: Очередь, Worker, Transcriber, Bot, Платежи, Тестирование, Cleanup — все они соответствуют ключевым этапам НИОКР
- **Бюджет часов**: 600h миграции хорошо вписывается в общий бюджет ~2470h проекта

### ⚠️ Что требует внимания

1. **TASK-16 (Bot Refactor, 140h)** — это ~23% всех часов миграции; требует особого контроля качества и регрессионного тестирования
2. **Параллелизм**: TASK-15, TASK-16, TASK-17 можно частично выполнять параллельно (месячные интервалы 5–6 НИОКР этапа 3), но при наличии одного разработчика последовательность мо́жет быть критична
3. **Post-release (этап 6 НИОКР)**: После TASK-22 требуется подготовка научно-технического отчёта, что может быть отдельным трудоёмким процессом (не учтено в 600h)

### 🎯 Рекомендации

1. **Укрепить CI с самого начала** (TASK-20 можно начать параллельно TASK-14)
2. **Подготовить staging окружение** на месячном интервале 7–8 (начало фазы 3) для ранней валидации
3. **Документирование результатов** начать с конца месячного интервала 9–10 (фаза 4) для подготовки итогового НТО
4. **Резервировать буфер** из месячного интервала 11–12 для unexpected issues и fine-tuning перед финальным релизом
