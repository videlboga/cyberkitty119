# Мета-инструкция: как использовать шаблоны документации

Этот файл описывает практику и процесс работы с документами в папке `docs/` и `docs/templates/`.

Цель: обеспечить единообразие, прослеживаемость и оперативность при принятии архитектурных и операционных решений.

1) Где лежат шаблоны
- `docs/templates/` — набор шаблонов: ADR, runbook, incident, migration plan, API contract, PR template и т.д.

2) Общие правила оформления
- Все документы должны начинаться с метаданных (YAML-like header): title, author, date, status, tags, related.
- Формат времени — ISO (UTC): 2026-01-23T06:19:00Z.
- Теги — kebab-case: `transcribe`, `infra`, `tg-bot`.

3) Жизненный цикл документа
- Draft: документ создан, в репозитории, но ещё обсуждается.
- Proposed: документ прошёл предварительное обсуждение, готов к обзору архитекторов/овнеров.
- Accepted: решение принято и используется в планах реализации.
- Deprecated: документ неактуален (ссылка на заменяющий ADR).

4) Процесс создания и утверждения
- Шаг 1 — создайте документ по шаблону в `docs/adr/` или соответствующей папке и привяжите к issue/ticket.
- Шаг 2 — отправьте PR с описанием изменений и ссылками на связанные тикеты.
- Шаг 3 — на PR должен быть хотя бы 1 review от другого разработчика и 1 review от архитектора/owner для ADR/infra.
- Шаг 4 — после одобрения обновите статус в метаданных на `Accepted` и при необходимости добавьте runbook / migration plan.

5) Инциденты и postmortem
- В случае инцидента создайте `docs/incidents/YYYY-MM-DD-<short-title>.md` по шаблону, заполните timeline и root cause в течение 24 часов.
- Проведите postmortem meeting и добавьте action items с owner'ами и ETA.

6) Документы и PR
- PR, меняющий infra/arch/production-critical код, должен ссылаться на ADR или включать новый ADR в том же PR.
- PR без ADR для infra изменений — не мержить (policy).

7) Автоматизированная валидация (CI)
- В CI pipeline добавить шаги:
  - markdownlint для стиля
  - Проверка метаданных (наличие title/author/date/status)
  - linkcheck для внутренних ссылок

8) Кто отвечает за docs
- Назначьте 1-2 maintainers (docs owners) в команде. Они проверяют PR/ADR и проводят периодические ревью.

9) Примеры быстрых шагов
- Создать ADR: `docs/adr/ADR-2026-001-simplify-ingest.md` по шаблону и открыть PR.
- Создать runbook для нового сервиса: `docs/runbooks/service-name-runbook.md`.

10) Полезные команды
- Просмотреть шаблоны: `ls docs/templates`
- Проверить метаданные: `grep -R "^---" -n docs | sed -n '1,50p'`

11) FAQ — короткие ответы
- Q: Кто может создать ADR? A: Любой разработчик, но требуется review от архитектора/owner.
- Q: Как перевести документ в Accepted? A: Через PR + approvals; затем обновить `status` в метаданных.
