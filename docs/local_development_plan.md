# План локальной разработки

Цель: безопасно и итеративно восстановить и проверить пайплайн загрузки → кеширования → скачивания → постановки в очередь → транскрибирования локально, обеспечить удобную и воспроизводимую среду для разработчиков и покрыть критичные места тестами.

Краткое содержание действий (в порядке выполнения):

1. Подготовка окружения разработчика (venv, локальные хуки, Docker-валидатор).
2. Воспроизведение ошибки локально: заставить `telegram-bot-api` поместить файл в том и воспроизвести копирование ботом.
3. Мониторинг и логирование: добавить краткие дополнительные лог-сообщения в `bot` и `worker` для диагностики путей файла и ошибок сети.
4. Тестирование downloader: написать юнит-тесты для `transkribator_modules/utils/large_file_downloader.py` (покрыть `_probe_cache_copy()` и fallback).
5. Интеграционный smoke-test: автоматически прогонять сценарий загрузки небольшого тестового файла и проверять запись в `processing_jobs` и создание `transcripts` (или тестовый мок).
6. Документация и pre-commit: обеспечить локальную валидацию docs и инструкции для разработчиков.

Предположения и ограничения
- Работа ведётся локально на машине разработчика с Docker (доступ к Docker требуется для некоторых шагов).
- Доступ к прод‑базе/продуктовым ресурсам не требуется — всё воспроизводится в dev/test окружении.
- Сетевые нюансы (WireGuard/IPv6) будут эмулироваться или временно отключаться в контейнерах при необходимости.

Контракт (что считается успехом)
- Маленький медиа-файл, загруженный через тестовый бот, появляется в общем томе `telegram-bot-api-data`.
- `bot` успешно обнаруживает файл, копирует его в `bot`-том и создаёт запись в `processing_jobs`.
- `worker` (или тестовый мок) обрабатывает задачу и создаёт (либо мокирует) запись в `transcripts`.
- Юнит-тесты для `large_file_downloader` проходят.

Пошаговый план

Шаг 0 — подготовка (15–30 мин)
- Создать и активировать virtualenv (если вы не хотите использовать Docker для локальных зависимостей).

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt  # если нужно
```

- Включить локальные git-хуки (однократно):

```bash
git config core.hooksPath .githooks
```

- Для разработчиков с ограничениями pip: запустить `.github/scripts/run_metadata_validator_docker.sh` для проверки docs.

Шаг 1 — воспроизведение и проверка томов (30–60 мин)
- Запустить локальный docker-compose (dev) или те сервисы, которые нужны: `telegram-bot-api`, `bot`, `worker`, `postgres`.
- Убедиться, что том `./telegram-bot-api-data` смонтирован в сервис `telegram-bot-api`.
- Отправить тестовое медиа в бот (можно использовать тестовый скрипт или curl к тестовому endpoint), либо скопировать заранее подготовленный файл в хостовую папку `telegram-bot-api-data/<BOT_TOKEN>/videos/`.

Проверки:
- `ls -l ./telegram-bot-api-data/*/videos` — файл виден.
- `docker compose logs telegram-bot-api` — проверяем, что tgapi успешно скачал файл.

Шаг 2 — локальное логирование и трассировка (15–60 мин)
- Добавить (временное) подробное логирование в `transkribator_modules/bot/handlers.py` и `transkribator_modules/utils/large_file_downloader.py` вокруг точек:
  - где вызывается `download_large_file`
  - что возвращает `_probe_cache_copy()`
  - в каких путях производится копирование
- Пересобрать image `bot` (или перезапустить контейнер) и воспроизвести сценарий.

Шаг 3 — юнит-тесты для downloader (1–2 ч)
- Создать `tests/test_large_file_downloader.py` с тестами:
  - `_probe_cache_copy()` возвращает путь когда есть ровно один подходящий файл
  - корректная обработка отсутствия expected size (поведение fallback)
  - симуляция прав доступа/ownership

- Использовать pytest и tmp_path для создания временных директорий и файлов.

Пример тест (скетч):

```python
def test_probe_cache_single(tmp_path):
    # создать структуру тома и один файл
    # вызвать функцию и проверить что путь возвращён
    assert True
```

Шаг 4 — интеграционный smoke-test (1–2 ч)
- Написать скрипт `scripts/smoke_test_pipeline.sh` который:
  - помещает тестовый файл в `telegram-bot-api` том
  - ждёт появления копии в `bot` том
  - проверяет запись в БД `processing_jobs`
- Этот скрипт можно запускать в CI как smoke test.

Шаг 5 — улучшения и мелкие рефакторы (по мере готовности)
- Вынести повторяющиеся проверки сети/egress в диагностический модуль.
- Подумать о миграции кеша в S3 (интерфейс storage abstraction) — проектная задача.

Приёмы диагностики (короткая шпаргалка)
- Проверить интерфейсы и маршруты в контейнере `telegram-bot-api`:

```bash
docker compose exec telegram-bot-api ip addr
docker compose exec telegram-bot-api ip route
docker compose exec telegram-bot-api wg show || true
```

- Для временного обхода WireGuard в тестовой среде:

```bash
docker compose exec telegram-bot-api ip link set wg0 down || true
```

- Мониторинг логов:

```bash
docker compose logs -f bot
docker compose logs -f worker
```

Критерии принятия (Done)
- Все тесты в `tests/` проходят локально.
- Smoke-test сценарий завершился успешно (создана запись в `processing_jobs`).
- Документация (`docs/developer-setup.md`, `docs/local_bot_api_download.md`, `docs/local_development_plan.md`) обновлена и валидируется скриптом.

Риски и откаты
- Если в ходе экспериментов будут изменены миграции БД — подготовить rollback script и тестовую базу.
- Любые изменения в production-конфигурации (wg0, routing) не делаются сразу в prod; тестировать локально и в staging.

План работ по дням (ориентир)
- День 0 (сейчас): подготовка окружения, включение хуков, проверить docs-валидатор.
- День 1: воспроизведение проблемы локально, базовые лог-уведомления.
- День 2: написать юнит-тесты и smoke-test; автоматизировать запуск.
- День 3: мелкие рефакторы, подготовка PR и документация.

Назначить ответственных и owners
- Автор изменений (dev): вы (или тот, кто откроет PR).
- Reviewer: 1 разработчик + 1 эксперт по infra/arch для финального мержа.

Следующие шаги (сейчас)
1. Запусти локально: `git config core.hooksPath .githooks`.
2. Активируй `.venv` или используй Docker-валидатор для docs.
3. Выбери задачу из списка (я могу начать с пункта 1 или 2 по твоему указанию).
